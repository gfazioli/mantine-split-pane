Comparison with Mantine core Splitter

Since version 9.3, Mantine core ships a first-party Splitter component, built on the headless use-splitter hook. The basics overlap — resizable panes, keyboard resizing, full Styles API — but the two make different trade-offs.

CapabilitySplitCore Splitter

Fully themeable standalone resizer

7 variants, colors, gradients, hover states, knob, custom cursors and custom content — core offers lineSize, handleColor, handleIcon and the Styles API, but no standalone reusable resizer, variants or gradients

Snap points

snapPoints, snapTolerance, snapFrom and the onSnap callback

Double-click reset

Both restore panes by double-clicking the resizer — Split to each pane's initialWidth / initialHeight, core to its default ratio (on by default since Mantine 9.3.2, toggle via resetOnDoubleClick)

Sizing in any CSS unit

px, %, rem for sizes and min/max constraints — core uses percentages summing to 100 (proportional by design)

Responsive orientation

orientation accepts Mantine breakpoint maps, e.g. base horizontal, md vertical

Responsive sizing values

Pane sizes, min/max, resizer size and snapPoints accept Mantine breakpoint maps

Config-driven layouts

Split.Dynamic generates panes and resizers from a PaneConfig array

Container resize tracking

Pane ratios are preserved when the container resizes

Keyboard resizing

Both: arrows with step and shiftStep — core also supports Home / End and Enter to collapse; Split adds Escape to cancel

Resize lifecycle events

Split: per pane and per resizer, fired on drag, keyboard and double-click — core: onResizeStart / onResizeEnd (handleIndex, sizes) on pointer drag only, while onSizeChange fires on any size change (drag, keyboard, double-click reset, collapse)

Imperative API via ref

Split: per-pane handlers (reset, size getters) — core: whole-splitter splitterRef (sizes, collapse, expand)

Collapsible panes

collapsible, collapseThreshold, onCollapseChange, imperative collapse and expand

Controlled sizes

sizes and onSizeChange props for state-driven layouts — Split is uncontrolled, DOM-driven

Space redistribution strategies

nearest, equal or a custom function across non-adjacent panes

Screen-reader semantics

WAI-ARIA separator pattern: role, aria-orientation, aria-valuenow / min / max — Split's resizer exposes an aria-label only

RTL support

Honors DirectionProvider: drag and arrow keys invert in right-to-left layouts

Headless hook

use-splitter for fully custom UIs

When to use which. Reach for the core Splitter when you need a simple proportional layout with first-party support — especially if collapsible panes, controlled sizes, space redistribution or a headless hook matter to you. Reach for Split when the resizer look and feel is part of your design (variants, gradients, knobs, custom content), when you need snap points, pixel-based constraints or responsive breakpoint-driven layouts, or when panes are generated dynamically from configuration.

Responsive Sizing

You can set pane size props (initialWidth, initialHeight, minWidth, maxWidth, minHeight, maxHeight) to be responsive by passing an object with breakpoints as keys. This uses the same pattern as orientation. Resize the browser window to see the pane sizes change based on the viewport size.

Additionally, panes with percentage-based sizes (e.g. initialWidth="50%") will automatically recalculate proportionally when the container is resized. If you manually drag a resizer and then resize the container, the drag position is preserved as a ratio and scales proportionally.

Gradient variant

When variant prop is set to gradient, you can control gradient with gradient prop, it accepts an object with from, to and deg properties. If the gradient prop is not set, Split.Resizer will use theme.defaultGradient which can be configured on the theme object. gradient prop is ignored when variant is not gradient.

Note that variant="gradient" supports only linear gradients with two colors. If you need a more complex gradient, then use Styles API to modify Split.Resizer styles.

Auto Resizers

Starting from version 2.6, you can use the autoResizers prop to automatically insert Split.Resizer components between Split.Pane components. This simplifies the JSX structure and makes the code cleaner, especially when working with multiple panes or dynamic layouts.

When autoResizers is set to true, you only need to define the Split.Pane components, and the resizers will be automatically inserted between them.

Snap points

You can make a resizer snap to common pane sizes by setting snapPoints and snapTolerance. These props are available on both Split and Split.Resizer.

  • Split provides default snap behavior to all child resizers.
  • Split.Resizer can override those values for a specific divider.
  • When using autoResizers, the generated resizers inherit snapPoints and snapTolerance from Split.

By default, snap points are interpreted as the size in pixels of the pane before the resizer: the left pane in vertical mode and the top pane in horizontal mode.

Drag a resizer near 200, 400, or 600 pixels to snap. The first split uses snapPoints inherited from Split , including autoResizers .

Percentages, responsive sets, snap from "after", and onSnap callback

snapPoints accepts percentage strings (e.g. '50%', resolved against the combined size of the two adjacent panes) as well as Mantine breakpoint maps for responsive snap sets:

<Split autoResizers snapPoints={['25%', '50%', '75%']}>
  <Split.Pane initialWidth={300}>Primary content</Split.Pane>
  <Split.Pane>Secondary content</Split.Pane>
</Split>

<Split autoResizers snapPoints={{ base: [200, 400], md: [300, 500, 700] }}>
  <Split.Pane initialWidth={300}>Primary content</Split.Pane>
  <Split.Pane>Secondary content</Split.Pane>
</Split>

Set snapFrom="after" to interpret each value as the target size of the pane after the resizer (useful when you want a panel to collapse to a fixed width on the right or bottom edge):

<Split autoResizers snapPoints={[200, 400]} snapFrom="after">
  <Split.Pane initialWidth={300}>Primary content</Split.Pane>
  <Split.Pane>Secondary content</Split.Pane>
</Split>

The onSnap callback on Split.Resizer is fired once when the resizer enters a snap zone (with the pixel value of the snap point, expressed in the snapFrom reference) and once when it leaves (with null). It is not fired on every pointer tick.

<Split.Resizer
  snapPoints={[200, 400, 600]}
  onSnap={(point) => {
    if (point !== null) {
      console.log('Snapped to', point);
    }
  }}
/>

While the resizer is inside a snap zone, the root element exposes a data-snapping attribute. You can target it in CSS to highlight the divider, show a marker, or play a subtle transition.

onResizeStart, onResizing, and onResizeEnd events

The Split.Resizer provides the same events onResizeStart, onResizing, and onResizeEnd that are available in the Split.Pane. However, pay attention to the difference in the parameter found in onResizing and onResizeEnd. In the case of the Split.Resizer, the parameter is an object with the properties width and height for both panes; the one on the left (or top) beforePane and the one on the right (or bottom) afterPane.

Since v4.1.2, onResizing and onResizeEnd also fire once after a double-click reset with the post-reset sizes, so consumers tracking pane dimensions don't need a separate onDoubleClick handler to react to the reset.

export type SPLIT_PANE_RESIZE_SIZES = {
  beforePane: {
    width: number;
    height: number;
  };
  afterPane: {
    width: number;
    height: number;
  };
};

The events released are therefore:

export interface SplitResizerProps {
  /** Event called when resizer is double clicked */
  onDoubleClick?: (
    event: React.MouseEvent<HTMLButtonElement>
  ) => void;

  /** Event called when pane size starts changing */
  onResizeStart?: () => void;

  /** Event called when pane size changes */
  onResizing?: (sizes: SPLIT_PANE_RESIZE_SIZES) => void;

  /** Event called when pane size changes */
  onResizeEnd?: (sizes: SPLIT_PANE_RESIZE_SIZES) => void;
}

          onResizeEnd={(sizes) => {
            setStart(false);
            setEnd(sizes);
          }}
          onResizing={setResizing}
        />

        <Split.Pane grow>
          <Paper withBorder w="100%" mih="100%">
            <Title>Pane 2</Title>
          </Paper>
        </Split.Pane>
      </Split>

      <Stack gap={2}>
        <Code>Start: {start ? 'true' : 'false'}</Code>
        <Code>
          End (beforePane): w={end?.beforePane.width} h={end?.beforePane.height}
        </Code>
        <Code>
          End (afterPane): w={end?.afterPane.width} h={end?.afterPane.height}
        </Code>
        <Code>
          Resizing (beforePane): w={resizing?.beforePane.width} h={resizing?.beforePane.height}
        </Code>
        <Code>
          Resizing (afterPane): w={resizing?.afterPane.width} h={resizing?.afterPane.height}
        </Code>
      </Stack>
    </Stack>
}

onResizeStart, onResizing, and onResizeEnd events

The Split.Pane provides the same events onResizeStart, onResizing, and onResizeEnd that are available in the Split.Resizer. However, pay attention to the difference in the parameter found in onResizing and onResizeEnd. In the case of the Split.Pane, the parameter is an object with the properties width and height of the selected pane.

Since v4.1.2, onResizing and onResizeEnd also fire once after a double-click reset on the adjacent resizer, with the post-reset pane size. The pane's internal drag state is intentionally not updated by this emission, so the reset still persists across subsequent container resizes.

export type SPLIT_PANE_SIZE = {
  width: number;
  height: number;
};

The events released are therefore:

export interface SplitPaneProps {
  /** Event called when pane size starts changing */
  onResizeStart?: () => void;

  /** Event called when pane size changes */
  onResizing?: (size: SPLIT_PANE_SIZE) => void;

  /** Event called when pane size changes */
  onResizeEnd?: (size: SPLIT_PANE_SIZE) => void;

  /** Event called to reset initial size */
  onResetInitialSize?: (
    e: React.MouseEvent<HTMLButtonElement>
  ) => void;
}

Reset with Double Click

You can reset the size of the panes by double-clicking on the separator. The two adjacent panes snap back to their initialWidth / initialHeight values, and the pane's internal "has been dragged" tracking is cleared so the reset persists across subsequent container resizes.

Starting from v4.1.2, the double-click reset also fires the onResizing and onResizeEnd callbacks — both on the Split.Resizer (with { beforePane, afterPane } sizes) and on each adjacent Split.Pane (with its own { width, height }) — so consumers tracking pane dimensions can react to the reset without subscribing to onDoubleClick separately.

Split.Dynamic

Starting from version 2.7, the Split.Dynamic helper function provides a declarative way to create dynamic layouts using a configuration array. This is particularly useful when you need to manage panes programmatically, show/hide panes conditionally, or build complex layouts with dynamic content.

Basic Usage

The Split.Dynamic function accepts a PaneConfig array that defines each pane's properties and content. Use it inside Split children as a function call (not as a JSX component):

<Split>{Split.Dynamic({ panes })}</Split>

This approach is cleaner than manually managing conditional rendering with keys.

        <Paper withBorder w="100%" h="100%">
          <h1>Main Content</h1>
          <p>This pane grows to fill available space</p>
        </Paper>
      ),
    },
    {
      id: 'info',
      initialWidth: 250,
      content: (
        <Paper withBorder w="100%" h="100%">
          <h1>Info Panel</h1>
          <p>Additional information</p>
        </Paper>
      ),
    },
  ];

  return (
    <Split w="100%" h={400}>
      {Split.Dynamic({ panes })}
    </Split>
  );
}

Conditional Rendering

Use the filter prop to conditionally show or hide panes. This is perfect for collapsible sidebars, panels that appear based on user actions, or responsive layouts.

Main Content

Info Panel

import { useState } from 'react';
import { Split, PaneConfig } from '@gfazioli/mantine-split-pane';
import { Button, Group, Paper } from '@mantine/core';

function Demo() {
  const [showSidebar, setShowSidebar] = useState(true);
  const [showInfo, setShowInfo] = useState(true);

  const panes: PaneConfig[] = [
    {
      id: 'sidebar',
      initialWidth: 200,
      content: (
        <Paper withBorder w="100%" h="100%">
          <h1>Sidebar</h1>
          <p>Collapsible sidebar</p>
        </Paper>
      ),
    },
    {
      id: 'main',
      grow: true,
      content: (
        <Paper withBorder w="100%" h="100%">
          <h1>Main Content</h1>
          <p>This pane is always visible</p>
        </Paper>
      ),
    },
    {
      id: 'info',
      initialWidth: 250,
      content: (
        <Paper withBorder w="100%" h="100%">
          <h1>Info Panel</h1>
          <p>Collapsible info panel</p>
        </Paper>
      ),
    },
  ];

  const filter = (pane: PaneConfig) => {
    if (pane.id === 'sidebar') return showSidebar;
    if (pane.id === 'info') return showInfo;
    return true;
  };

  return (
    <>
      <Split w="100%" h={400}>
        {Split.Dynamic({ panes, filter })}
      </Split>
      <Group mt="md" justify="center">
        <Button onClick={() => setShowSidebar((s) => !s)}>
          {showSidebar ? 'Hide' : 'Show'} Sidebar
        </Button>
        <Button onClick={() => setShowInfo((s) => !s)}>
          {showInfo ? 'Hide' : 'Show'} Info
        </Button>
      </Group>
    </>
  );
}

Advanced Dynamic Layouts

You can combine Split.Dynamic with state management to create fully dynamic layouts. In this example, panes can be added or removed dynamically, and each pane can have custom resizer props.

import { useState } from 'react';
import { Split, PaneConfig } from '@gfazioli/mantine-split-pane';
import { Button, Code, Group, Paper, Title } from '@mantine/core';

function Demo() {
  const [paneCount, setPaneCount] = useState(3);

  const panes: PaneConfig[] = Array.from({ length: 5 }, (_, i) => ({
    id: `pane-${i + 1}`,
    initialWidth: i === 2 ? undefined : 150,
    grow: i === 2,
    content: (
      <Paper withBorder w="100%" h="100%">
        <Title order={3}>Pane {i + 1}</Title>
        <p>{i === 2 ? 'Growing pane' : 'Fixed width'}</p>
      </Paper>
    ),
    resizerProps: i % 2 === 0 ? { variant: 'dotted' } : undefined,
  }));

  return (
    <>
      <Split w="100%" h={400}>
        {Split.Dynamic({
          panes,
          filter: (pane) => parseInt(pane.id.split('-')[1], 10) <= paneCount,
        })}
      </Split>
      <Group mt="md" justify="center">
        <Button
          onClick={() => setPaneCount((c) => Math.max(2, c - 1))}
          disabled={paneCount <= 2}
        >
          Remove Pane
        </Button>
        <Code>Active Panes: {paneCount}</Code>
        <Button
          onClick={() => setPaneCount((c) => Math.min(5, c + 1))}
          disabled={paneCount >= 5}
        >
          Add Pane
        </Button>
      </Group>
    </>
  );
}

Note well

The PaneConfig interface supports all props from Split.Pane (such as initialWidth, grow, minWidth, maxWidth, etc.) plus an optional resizerProps field to customize the resizer that appears after each pane. Each pane must have a unique id prop.

interface PaneConfig {
  id: string;
  content: React.ReactNode;
  // All Split.Pane props
  initialWidth?: number | string;
  grow?: boolean;
  minWidth?: number | string;
  // ... other Split.Pane props
  // Optional custom resizer props
  resizerProps?: Omit<SplitResizerProps, 'beforeRef' | 'afterRef'>;
}

Accessibility

You may also use the Keyboard to navigate through the panes. Use the Tab key to navigate through the resizer and the Arrow keys to resize the panes. You may use ◀︎ ▶︎ for horizontal panes and ▼ ▲ for vertical panes.

By default, the Arrow keys will resize the panes by 8 pixel, but you can change this value using the step prop. Keeping the SHIFT key pressed while using the Arrow keys will resize the panes by 64 pixel, but you can change this value using the shiftStep prop.

import { Paper } from '@mantine/core';

function Demo() {

        <Paper {...paperProps}>

        </Paper>

Note well

The step and shiftStep props are accessible both for the Split and Split.Resizer components. Obviously, the step and shiftStep props of the Split.Resizer component will override the step and shiftStep props of the Split component.

<Split step={16} shiftStep={128}>
  <Split.Pane>
    <h1>Pane 1</h1>
  </Split.Pane>

  <Split.Resizer step={32} shiftStep={256} />

  <Split.Pane>
    <h1>Pane 2</h1>
  </Split.Pane>

  <Split.Resizer />

  <Split.Pane>
    <h1>Pane 3</h1>
  </Split.Pane>
</Split>

In the above example, the resizer between the Pane 1 and Pane 2 will resize the panes by 32 pixel, and the resizer between the Pane 2 and Pane 3 will resize the panes by 16 pixel. The SHIFT key will resize the panes by 256 pixel between the Pane 1 and Pane 2 and by 128 pixel between the Pane 2 and Pane 3.
